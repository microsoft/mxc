# WSLC SDK Bindings ŌĆö bindgen Migration & SDK-Update Runbook

The WSLC backend calls the WSLC SDK (`wslcsdk.dll`) through a C ABI. The Rust
FFI bindings for that ABI are **generated by [bindgen](https://rust-lang.github.io/rust-bindgen/)**
from the SDK's C header (`include/wslcsdk.h`), not hand-written. This makes
struct layout, sizes, function signatures, and enum values derive directly from
the header, so **any ABI drift becomes a compile error instead of silent
undefined behavior**.

## File layout

| File | Role |
|---|---|
| `src/backends/wslc/common/src/wslcsdk_sys.rs` | **Generated** by bindgen. The single source of ABI truth: opaque settings structs + size consts, handle types, enums, data structs, callback typedefs, and the runtime-loaded `WslcSdk` function table with compile-time size/offset asserts. **Do not hand-edit.** |
| `src/backends/wslc/common/src/wslc_bindings.rs` | Thin ergonomic **fa├¦ade** over the generated module. Re-exports the generated surface (`pub use crate::wslcsdk_sys::*;`) and adds only what bindgen cannot generate: the `S_OK` COM sentinel, `WslcSdk::load()` (anti-hijack DLL loading from the executable's own directory), the RAII guards (`WslcSessionGuard` / `WslcContainerGuard` / `WslcProcessGuard`), `check_hresult`, and `WslcComponentFlags::any_missing`. |
| `scripts/generate-wslc-bindings.ps1` | Regenerates `wslcsdk_sys.rs` from the header vendored in the SDK `.nupkg`. Run this on every SDK version bump. |

The generated file is **committed** to the repo. Normal builds (including CI and
the `--features wslc` feature build) simply `include`/`mod` the committed file ŌĆö
they need **no** libclang and **no** bindgen in the Cargo graph. bindgen and
libclang are required **only** on the machine that regenerates the bindings
after an SDK bump.

### Why a committed file + standalone script (not `build.rs`)

bindgen is deliberately kept out of `Cargo.toml` and `build.rs`. Generating at
build time would force **every** builder of the `wslc` feature to install
libclang and pull `bindgen`/`clang-sys` into the dependency graph (and into the
`MxcDependencies` feed). Committing the generated file keeps normal builds
dependency-free and deterministic; only SDK bumps touch the toolchain.

## Regeneration prerequisites

Needed **only** when regenerating bindings (i.e., on an SDK version bump):

- **LLVM / libclang** ŌĆö e.g. install LLVM (`C:\Program Files\LLVM\bin`). bindgen
  parses the C header via libclang.
- **`bindgen-cli`** ŌĆö `cargo install bindgen-cli` (installs `~/.cargo/bin/bindgen.exe`).

`scripts/generate-wslc-bindings.ps1` auto-discovers both, plus the MSVC/Windows
SDK include paths and the header inside the vendored `.nupkg`.

## Runbook ŌĆö updating to a new WSLC SDK version

Do all of the following in a **single commit** so the checked-in bindings always
match the pinned SDK.

1. **Bump the pinned version.** Update `WSLC_SDK_VERSION` in
   `src/backends/wslc/common/build.rs`.

2. **Pin the new package hash.** Add a match arm for the new version in
   `expected_sha256()` in `build.rs` with the new `.nupkg`'s lowercase-hex
   SHA-256. Without a pinned hash the feature build refuses the package
   (`no pinned SHA-256 for WSLC SDK v<version>`). During bring-up you can pass
   `WSLC_SDK_SHA256` via env to discover the hash, but the final commit must
   pin it in `expected_sha256()`.

3. **Refresh the vendored package** *(only while the `external/wslc-sdk/`
   fallback is still in use)* ŌĆö replace the vendored `.nupkg` with the new
   version so offline/fallback builds match.

4. **Regenerate the bindings.**

   ```powershell
   .\scripts\generate-wslc-bindings.ps1
   ```

   This extracts `include/wslcsdk.h` from the new `.nupkg` and rewrites
   `src/backends/wslc/common/src/wslcsdk_sys.rs`.

5. **Build ŌĆö this is where drift surfaces.**

   ```powershell
   cd src
   cargo build -p wslc_common
   ```

   If the header changed struct layout, sizes, signatures, or enum names, one of
   the following now **fails to compile** (instead of corrupting memory at
   runtime):
   - the generated compile-time size/offset asserts in `wslcsdk_sys.rs`;
   - the fa├¦ade in `wslc_bindings.rs` (e.g. a changed release signature no
     longer matches the `*ReleaseFn` aliases);
   - the runner in `wsl_container_runner.rs` (renamed struct fields, enum
     variants, or callback signatures).

   Fix the fa├¦ade and runner call sites to match the regenerated names/types.

6. **Validate.**

   ```powershell
   cd src
   cargo test -p wslc_common      # includes ABI size/enum asserts
   cargo fmt -p wslc_common -- --check
   cargo clippy -p wslc_common --all-targets -- -D warnings
   ```

7. **Run the WSLC E2E suite on a live WSL host** (cannot run in GitHub Actions ŌĆö
   needs WSL2 + the WSLC runtime). Build with the feature so `wslcsdk.dll` is
   staged next to `wxc-exec.exe`, then:

   ```powershell
   cargo build -p wxc --features wslc --release --target x86_64-pc-windows-msvc
   .\tests\scripts\run_wslc_all_tests.ps1
   ```

   Expect all configs to pass (the `quay.io` pre-pull and the two tar-import
   tests are environment-dependent and may skip).

8. **Commit** the regenerated `wslcsdk_sys.rs` **together with** the
   `WSLC_SDK_VERSION` + `expected_sha256()` change (and any fa├¦ade/runner fixes)
   in one commit.

## Contrast with the old hand-written bindings

Before bindgen, `wslc_bindings.rs` was a hand-written FFI table pinned to one
specific SDK ABI. A header change you didn't notice ŌĆö a struct that grew, a
renamed export, a reordered field ŌĆö produced **silent memory corruption or a
load failure at runtime**. For example, the 2.8.1 ŌåÆ 2.9.3 bump changed struct
sizes, renamed `WslcCanRun` ŌåÆ `WslcGetMissingComponents`, renamed
`CurrentDirectory` ŌåÆ `WorkingDirectory`, and added a `registryAuth` field ŌĆö all
of which had to be found and fixed by hand.

With bindgen, the same class of change **cannot compile** until a human
regenerates the file (reading the new header through bindgen) and fixes the
callers. Drift is converted from a runtime hazard into a build-time error.
